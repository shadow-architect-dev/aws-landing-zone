# AWS Landing Zone プラットフォームサービス SLI/SLO 定義書

本ドキュメントは、プラットフォームエンジニアリングチームが全社（各開発チーム・システム）に向けて提供している「共通ネットワーク」および「共通セキュリティ・基盤サービス」について、その品質を保証・維持するためのサービスレベル指標 (SLI) およびサービスレベル目標 (SLO) を定義したものです。

---

## 🏗️ 1. ネットワーク ＆ 通信品質 (Network & Egress)

### 1.1 共通 Egress ネットワーク可用性
* **背景**: 全アカウントのアウトバウンドインターネット通信を集約する Transit Gateway (TGW) および集約 NAT Gateway の可用性は、全アプリの外部 API 連携等の生命線です。
* **SLI (測定指標)**:
  - `(1 - (TGW/NAT GW でのパケットドロップ数 / 送信総パケット数)) * 100`
* **SLO (目標値)**:
  - **`月間 99.99% 以上`** (パケットドロップ率が月間 0.01% 以下であること)
* **SRE アクション**: パケットドロップが閾値を超えた場合、集約 NAT Gateway の帯域幅制限（ポート枯渇）を検知し、スケールアウト（IP追加）を自動トリガーします。

### 1.2 IPAM アドレス空間の残存容量 (可用性・拡張性)
* **背景**: VPC IPAM によって IP を動的割り当てしていますが、アドレス空間が枯渇すると、新規 VPC や新規環境の構築が完全に停止します。
* **SLI (測定指標)**:
  - `(IPAM 親プール内の利用可能な残存IPアドレス数 / 親プールの総IPアドレス数) * 100`
* **SLO (目標値)**:
  - **`常時 20% 以上`** の未割り当てIPアドレス空間（バッファ）を確保すること
* **SRE アクション**: 残存バッファが 20% を下回った場合、Datadog/CloudWatch アラートを発報し、IPAM に新しい CIDR ブロックを拡張アタッチする計画プロセスを開始します。

## 🔑 2. アイデンティティ ＆ セキュリティ (Identity & Security)

### 2.1 認証プラットフォーム可用性
* **背景**: IAM Identity Center (SSO) を通した AWS マネジメントコンソールや CLI へのアクセスが遮断されると、全社の開発・運用業務が全面的に停止します。
* **SLI (測定指標)**:
  - `(AWS SSO Portal へのログイン成功数 / SSO Portal へのログイン試行総数) * 100`
* **SLO (目標値)**:
  - **`月間 99.99% 以上`**
* **SRE アクション**: ログイン失敗率が 0.01% を超えた場合、認証プロバイダ (Google Workspace) と AWS 間の SAML/OIDC 連携エラー、または IdP 側のサービスダウンと判定し、緊急アクセス（Break-Glass）の適用を検討します。

### 2.2 セキュリティ・ガバナンス準拠率
* **背景**: メンバーアカウントでガードレールやセキュリティベースラインから外れた設定の「ドリフト」が発生していないかを継続監査します。
* **SLI (測定指標)**:
  - `AWS Security Hub 組織ダッシュボードで評価される「Security Score (%)」の加重平均値`
* **SLO (目標値)**:
  - **`常時 95% 以上`** のセキュリティスコアを維持すること
* **SRE アクション**: スコアが 95% 未満となった場合、または「重要度: HIGH/CRITICAL」の非準拠リソースが 3 日以上放置された場合、対象アカウントの管理チームへ自動警告チケットを発行し、強制適用プロセスへ移行します。

## 📊 3. ログ ＆ 可観測性 (Logging & Observability)

### 3.1 ログ集約パイプライン遅延 (Latency)
* **背景**: セキュリティインシデントや障害発生時の調査において、CloudTrail やコンテナログの転送遅延は監査不備や初動対応の遅れに直結します。
* **SLI (測定指標)**:
  - `各メンバーアカウントでのログイベント発生時間と、Log Archive アカウントの S3 バケットにログが書き込まれた時間の差（99パーセンタイル）`
* **SLO (目標値)**:
  - ログの 99% が **`5分 以内`** に Log Archive バケットへ転送完了すること
* **SRE アクション**: 転送遅延が 5分 を超過した場合、ログ集約用の Kinesis Firehose や Fluent Bit コンテナの詰まり（スロットリング）とみなし、バッファサイズ調整やコンテナの自動リスタートを実行します。

### 3.2 可観測性データ転送遅延 (メトリクス Latency)
* **背景**: メトリクスの転送に遅延があると、監視ダッシュボードの表示が遅れ、障害検知や自動アラートの発報遅延（MTTRの悪化）を招きます。
* **SLI (測定指標)**:
  - `アプリケーション・インフラでイベントが発生した時刻と、Datadog 側にデータポイントが記録された時刻の差`
* **SLO (目標値)**:
  - メトリクスデータの 99% が **`1分 以内`** に Datadog に到達すること

## ⚡ 4. 運用効率 ＆ 開発者体験 (Operation & DX)

### 4.1 AFT アカウントプロビジョニング所要時間 (Duration)
* **背景**: 新規プロジェクトが発足した際、AWS アカウントが即座に使えることは開発スピード（デベロッパー体験）の向上に不可欠です。
* **SLI (測定指標)**:
  - `accounts.yaml への追加コミットの main ブランチマージから、AFT 管理モジュールが AWS アカウントを払い出し、全セキュリティベースライン（SecurityHub/Datadog統合等）のデプロイが完了するまでの総時間（90パーセンタイル）`
* **SLO (目標値)**:
  - アカウント申請の 90% 以上が **`30分 以内`** に完了すること
* **SRE アクション**: アカウント構築が 30 分を超過した場合、Control Tower Service Catalog のスロットリングまたは AWS Organizations の上限緩和（Quotas）の上限到達とみなし、AWSサポートへ自動チケット連携を行います。
